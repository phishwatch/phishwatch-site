The problem

Why existing defenses
miss browser-native attacks

Traditional phishing defenses operate at delivery — scanning emails before they reach your inbox. But modern attacks activate after delivery, through legitimate cloud services and browser mechanics that no email filter can observe.

Email filters check URLs against blacklists at delivery time. Google Safe Browsing and SmartScreen assess domain reputation. None of these tools can observe what happens inside your browser after you click — the redirect chains, the credential flow, the copy-paste-execute sequence.

The browser is the attack surface. Recent research identified 287 Chrome extensions across 37M+ installs quietly transmitting user browsing history (SecurityWeek, CSO Online). This reinforces a structural reality: trust cannot be assumed from store approval, HTTPS, or domain reputation alone. Runtime mechanics are what matter — and that is where PhishWatch operates.

Static trust signals are brittle. Runtime mechanics are durable. PhishWatch evaluates what an action does, not what a page claims to be.

Structural thesis

Why runtime is the
last defensible layer

Traditional defenses operate at content or reputation layers — both of which attackers can adapt around:

Reputation systems are reactive

Blocklists and domain age checks lag behind attack infrastructure. New domains and throwaway hosts bypass reputation entirely.

AI content classification is bypassable

Adaptive wording, brand mimicry, and LLM-generated variations erode semantic detection. Content is malleable.

Attackers change presentation

Infrastructure, copy, logos, and sender domains are all replaceable. Static trust signals break under adaptive adversaries.

Mechanics are non-negotiable

To steal credentials, attackers must execute browser mechanics. They cannot complete the attack without navigating, submitting, or executing — observable actions.

PhishWatch operates at the mechanic layer — where the attack must succeed to matter.

This creates a growing corpus of mechanic-level signals that cannot be reverse-engineered from content alone — a structural defensibility that increases with deployment scale.

Detection methods

Runtime detection of
risky mechanics

PhishWatch evaluates mechanics — redirects, credential flow, interaction coupling — not whether a page looks legitimate. We detect patterns in how attacks work, not what they say.

ClickFix Detection

Detects copy events and correlates them with near-term outbound navigation in execution-lure contexts. PhishWatch may inspect text you copy locally on-device to detect command or OAuth-code lure patterns, but clipboard text is never transmitted or stored.

copy→navigate coupling

AiTM Detection

Observes redirect chains and detects credential submission to origins that do not match the page origin or expected authentication flow. Flags MFA-bypass proxying patterns structurally.

redirect chain origin mismatch

Browser-in-the-Browser

Detects DOM overlay patterns and origin-context anomalies consistent with browser-window spoofing techniques. Flags suspicious window overlays without requiring content inspection.

DOM overlay identity mismatch

ConsentFix Detection

Detects OAuth token hijacking via paste events. PhishWatch may inspect pasted text locally on-device to identify authorization code patterns, then blocks paste if coupled with suspicious navigation. Clipboard text is never transmitted or stored.

paste→navigate blocking

Process

How detection works

Five steps, all transparent. User remains in control at every point.

01

Click

User clicks a link. Normal browsing — no interruption yet.

02

Local analysis

Copy/paste patterns, redirect chains, credential flow analysed in-browser. No data leaves your device at this stage.

03

Cloud check

If local signals suggest risk, the destination URL plus sanitised signal metadata is sent for scoring. No page content, no clipboard data.

04

Risk score

Local and cloud signals are combined into a risk assessment with a specific mechanical reason.

05

Warn + override

User sees a specific, explainable warning and can proceed anyway. PhishWatch never blocks unconditionally.

When PhishWatch activates

PhishWatch doesn't scan every click — only those with risk indicators. Activation happens when:

  • Suspicious outbound links
    Punycode domains, IP addresses, URL shorteners, suspicious TLDs, credential-related keywords in URLs, or unusual redirect parameters
  • Sensitive authentication flows
    Credential input windows, password field interactions, or OAuth-like transitions where origin verification is critical
  • Copy/paste-to-execute patterns
    ClickFix and ConsentFix attack mechanics — text copy events followed by navigation showing copy→execute coupling patterns (e.g., PowerShell/cmd keywords, execution instructions)

Normal browsing on trusted domains without these indicators proceeds without interruption. These activation rules are continuously refined against real-world evasion attempts, creating a mechanic-level knowledge base that strengthens with scale.

Warning system

Explainable warnings,
not black boxes

Every PhishWatch warning includes a specific mechanical reason — not just "this site may be dangerous." Users see what triggered the warning and can make an informed decision.

This risk band system applies consistently across the browser extension overlay and all warning surfaces.

High

Strong mechanical evidence

e.g. ClickFix copy→navigate coupling detected; AiTM credential-flow mismatch; BitB DOM overlay with origin mismatch.

Medium

Suspicious pattern, lower confidence

e.g. Unusual redirect chain without confirmed credential submission; partial signal match.

Info

Informational — no action required

e.g. Known tracking redirect noted; no risky mechanics detected. Browsing continues uninterrupted.

Privacy architecture

Local-first detection.
Minimal cloud exposure.

PhishWatch performs local runtime checks in your browser first. Cloud scoring is only triggered when an outbound click shows risk indicators (e.g., suspicious URL structure, ClickFix patterns, credential flows). Most normal clicks aren't scanned — cloud checks activate only when needed.

What IS transmitted (when cloud check triggers)
  • Destination URL
    Required for backend risk assessment
  • Signal IDs & severity
    e.g. "instruction_to_execute_recent_copy", "high"
  • Timing metadata
    Timestamps, time-deltas between events (ms)
  • Boolean flags & field types
    e.g. paste_blocked: true, field_type: "password"
Activation policy: PhishWatch only contacts the cloud when an outbound click shows risk indicators (suspicious URL structure, ClickFix/ConsentFix patterns, credential flows, etc.). Most normal navigation doesn't trigger a scan — cloud checks activate only when local signals suggest risk.
What is NEVER transmitted
  • Page content or DOM structure
    Some checks inspect page text locally (on-device), but no page content or DOM is transmitted
  • Form fields, passwords, credentials
    Zero form data of any kind
  • Clipboard contents
    May be inspected locally on-device for ClickFix/ConsentFix patterns (e.g., command or OAuth-code lures), but never transmitted or stored
  • Cookies or session tokens
    No session data, no browsing history
  • User identifiers or tracking data
    No account linkage, no persistent user ID
Sanitisation is enforced in background.js via sanitizeClientSignals() — an allowlist-based function that strips any unlisted field before transmission. Unknown fields fail closed.
Data handling: PhishWatch does not store user accounts or persistent identifiers. Our backend processes requests in-memory. Standard server logs may record basic request metadata (e.g., timestamp and response status). Log retention is controlled by the deployment environment.

Scope

What PhishWatch
doesn't do

PhishWatch is a browser-layer runtime detection tool. It works best as part of a layered security strategy — complementing, not replacing, email filters, endpoint protection, and user awareness training.

Our focus: Runtime interruption of risky user transitions before credential or session compromise.
  • Stop social engineering persuasion or convince users not to click
  • Endpoint malware analysis or infection removal
  • Email content analysis or inbox filtering
  • Audit installed extensions or block third-party data sharing
  • Patch software vulnerabilities or OS-level exploits
  • Guarantee protection — no security layer can make this claim

FAQ

Common questions

How does PhishWatch differ from browser extension security tools?
PhishWatch focuses on runtime detection of phishing mechanics — copy-paste-execute patterns, credential-flow manipulation, fake browser windows — at the moment they occur. We do not audit installed extensions, block third-party data sharing, or provide extension governance. Recent incidents showing trusted extensions transmitting user data reinforce why runtime, action-aware detection is a critical security layer: static trust signals cannot evaluate what happens during actual user interactions.
What data does PhishWatch transmit?
PhishWatch performs local runtime checks first. Cloud scoring is only triggered when local signals suggest risk — not on every navigation. When triggered, we send the destination URL and sanitised signal metadata: signal IDs, severity levels, timing deltas, and boolean flags. Copied text, page content, form fields, passwords, cookies, and browsing history are never transmitted. See the Privacy section above for full details.
Will PhishWatch block legitimate sites or OAuth flows?
PhishWatch uses a fail-open design: when detection is uncertain, browsing continues without interruption. We reduce false positives on OAuth and SSO flows by requiring additional mechanical evidence beyond novelty (e.g., cross-origin credential submission, redirect chain anomalies). Local checks still run, but warnings only appear when multiple risk indicators align.
Can PhishWatch be used alongside existing security tools?
Yes. PhishWatch is designed to complement existing layers — email filters, endpoint protection, VPNs, password managers. We operate at the browser runtime layer and do not conflict with other tools.
What browsers are supported?
PhishWatch is currently available for Chrome and Chromium-based browsers. Support for additional browsers is planned.
Will PhishWatch slow down my browser?
No. Local analysis runs as lightweight event correlation. Cloud scoring is only triggered on suspicious transitions — not on normal navigation. There is no impact on everyday browsing performance.
Is PhishWatch available for enterprise deployment?
Yes. PhishWatch's architecture supports centralized policy configuration and telemetry-ready event streams (alpha webhook; SIEM integrations planned). Chrome Enterprise deployment is supported. We're currently running a pilot program for security teams. Contact security@phishwatch.io to discuss enterprise deployment, custom policy requirements, or telemetry integration.
