Structural thesis

Why runtime is the
last defensible layer

PhishWatch evaluates browser mechanics such as credential submission flows and redirect chains rather than relying on domain reputation or page content analysis. This is the architectural difference.

Reputation systems are reactive

Blocklists lag behind attack infrastructure. New domains and throwaway hosts bypass reputation entirely.

AI content classification is bypassable

Adaptive wording, brand mimicry, and LLM-generated variations erode semantic detection. Attacker-controlled AI-generated content bypasses these systems by design.

Attackers change presentation

Infrastructure, copy, logos, sender domains are replaceable. 66.9% more phishing emails now come from legitimate platforms that bypass authentication.

Mechanics are non-negotiable

To steal credentials, attackers must execute browser mechanics. They cannot complete the attack without navigating, submitting, or executing.

PhishWatch operates at the mechanic layer — where the attack must succeed to matter.

This creates a growing corpus of mechanic-level signals that cannot be reverse-engineered from content alone.

Detection methods

Runtime detection of
risky mechanics

Ten detection signals evaluate mechanics — redirects, credential flow, interaction coupling — not whether a page looks legitimate.

ClickFix Detection

Detects copy events and correlates them with near-term outbound navigation in execution-lure contexts. Clipboard text is inspected locally — never transmitted. North Korean APT groups and Scattered Spider have adopted ClickFix as a primary vector.

copy→navigate coupling

AiTM / Tycoon 2FA Detection

Observes redirect chains and detects credential submission to origins that don't match the expected flow. Detects browser-side AiTM mechanics: cross-origin credential POST and redirect chains preceding credential forms. Server-side session relay is invisible to browser extensions — PhishWatch detects the browser-observable mechanics of the attack.

redirect chain origin mismatch

Browser-in-the-Browser

Detects DOM overlay patterns and origin-context anomalies consistent with browser-window spoofing. Flags suspicious window overlays without requiring content inspection.

DOM overlay identity mismatch

ConsentFix Detection

Detects OAuth token hijacking via paste events. Inspects pasted text locally to identify authorization code patterns, then blocks paste if coupled with suspicious navigation.

paste→navigate blocking

Typosquatting Detection

Detects domains impersonating brands through character substitution, homoglyph swaps, and edit-distance analysis. Checked against a curated brand list — the same technique behind spoofed domains used in post-breach phishing campaigns.

brand similarity edit distance

Newly Registered Domain

Domains registered within 30 days are a strong indicator of attack infrastructure. PhishWatch checks domain age via RDAP (public ICANN protocol) and national WHOIS registries — no API key required. This is optional — if the query fails, detection continues without this signal. Research shows most parked domains now serve malicious content.

domain age < 30 days RDAP/WHOIS

Fake Update Detection

Detects pages impersonating browser update dialogs (SocGholish). Flags pages combining browser brand impersonation, update urgency language, and executable download links.

brand impersonation executable link

AI Lure Detection

Detects pages impersonating AI services (ChatGPT, Claude, Gemini, Copilot) combined with ClickFix or ConsentFix social engineering. Attackers use fake AI domains to trick users into running malicious commands.

AI brand spoof ClickFix lure

Process

How detection works

Five steps, all transparent. User remains in control at every point.

01

Click

User clicks a link. Normal browsing — no interruption yet.

02

Local analysis

Copy/paste patterns, redirect chains, credential flow analysed in-browser. No data leaves your device.

03

Cloud check

If local signals suggest risk, Domain name + sanitised signal metadata sent for scoring. No page content, no full URL.

04

Risk score

Local and cloud signals combined into a risk assessment with a specific mechanical reason.

05

Warn + override

User sees an explainable warning and can proceed anyway. Never blocks unconditionally.

Warning system

Explainable warnings,
not black boxes

Every PhishWatch warning includes a specific mechanical reason — not just "this site may be dangerous." Users see what triggered the warning and can make an informed decision.

Enterprise tools often produce opaque risk scores. PhishWatch produces simple explanations like: "This login form sends credentials to a different domain" or "You were redirected through multiple unknown domains."

High

Strong mechanical evidence

e.g. ClickFix copy→navigate coupling; AiTM credential-flow mismatch; BitB DOM overlay with origin mismatch.

Medium

Suspicious pattern, lower confidence

e.g. Unusual redirect chain without confirmed credential submission; partial signal match.

Low

Low confidence signal — no warning shown

e.g. Newly registered domain without additional signals; partial heuristic match below threshold. Scan is recorded in telemetry. Browsing continues without interruption.

Scope

doesn't do

  • Endpoint malware analysis or infection removal

FAQ

Common questions

How does PhishWatch differ from browser security platforms?
Browser security platforms like Seraphic and LayerX focus on enterprise policy enforcement, data loss prevention, and extension governance. PhishWatch focuses only on phishing mechanics at runtime: cross-origin credential submission, suspicious redirect chains, and deceptive login transitions. It protects the moment the user is about to act, not the browser environment itself.
Can PhishWatch detect Tycoon 2FA and similar AiTM attacks?
Yes. PhishWatch's AiTM detection identifies the structural mechanics all adversary-in-the-middle kits share: credential submission to origins that don't match the expected authentication flow, and cross-origin POST requests through transparent reverse proxies. This catches the attack class regardless of which kit is used. Even after the Tycoon 2FA takedown, the underlying technique persists.
Isn't phishing solved with MFA or passwordless login?
Passwordless authentication removes credential phishing, but not social-engineering risk. Modern attacks still rely on malicious OAuth consent flows, session-token theft, fake collaboration links, malware downloads, and crypto-wallet scams. PhishWatch addresses the human decision layer, which remains vulnerable even in passwordless environments.
What data does PhishWatch transmit?
Local runtime checks run first. Cloud scoring is only triggered when local signals suggest risk. When triggered, we send the destination domain name and sanitised signal metadata. Copied text, page content, form fields, passwords, cookies, and browsing history are never transmitted.
Will PhishWatch block legitimate sites or OAuth flows?
PhishWatch uses a fail-open design: when detection is uncertain, browsing continues without interruption. Warnings only appear when multiple risk indicators align. Users can always override.
Can PhishWatch be used alongside existing security tools?
Yes. PhishWatch complements email filters, endpoint protection, VPNs, and password managers. It operates at the browser runtime layer and does not conflict with other tools.
What browsers are supported?
Chrome and Chromium-based browsers. Additional browser support is planned.
Is PhishWatch available for enterprise deployment?
Yes. Chrome Enterprise deployment supported. Telemetry event stream built and live — five event types including session incident correlation. CEF webhook output for SIEM ingest today; native SIEM connectors on the roadmap. Contact [email protected] for pilot program details.
