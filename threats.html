01

ClickFix

Instruction-to-execute social engineering

What it is

ClickFix tricks users into copying a malicious command — typically PowerShell or a shell script — and pasting it into the Windows Run dialog, Terminal, or File Explorer address bar. The page presents a fake CAPTCHA, verification prompt, or error message that instructs the user to "fix" an issue by running a command. Because the user executes the command themselves, traditional endpoint protection often treats it as legitimate user activity.

Attack flow

1. Lure pageFake CAPTCHA, error, or verification prompt
2. Clipboard writeMalicious command via writeText() or user copy
3. PhishWatch detectsClipboard write intercepted, command patterns analysed locally
4. ExecutionUser opens Run/Terminal and pastes — blocked by warning

What PhishWatch observes

malicious_command_high_confidence

Clipboard write event containing PowerShell obfuscation patterns (encoded commands, Invoke-Expression, hidden window style), curl-pipe-to-bash chains, or certutil download commands. Inspected locally — clipboard text is never transmitted.

instruction_to_execute_recent_copy

Copy event followed by outbound navigation within a short time window, in a context where the page contains execution instructions (Win+R, paste, terminal). This coupling pattern is the mechanical signature of ClickFix.

02

ConsentFix

OAuth authorization code theft

What it is

ConsentFix exploits the OAuth authorization code flow. The attacker creates a fake login page that instructs the user to sign in with Google or Microsoft, copy the resulting authorization code, and paste it into a field on the attacker's page. This gives the attacker a valid OAuth token — bypassing MFA entirely because the user has already authenticated.

Attack flow

1. Fake loginPage mimics OAuth consent screen
2. User authenticatesReal OAuth flow completes, code generated
3. Code pasteUser pastes OAuth code into attacker's page
4. PhishWatch detectsOAuth code pattern in paste event on untrusted domain

What PhishWatch observes

consent_fix_paste_blocked

Paste event containing an OAuth authorization code pattern (4/0A..., code=...) into a credential field on a domain that is not a recognized OAuth provider. The paste is blocked before the code can be submitted. Pasted text is inspected locally and never transmitted.

PhishWatch also performs a passive page scan on load, detecting ConsentFix lure indicators: instructions to copy authorization codes, paste localhost URLs, or multi-step verification flows with paste instructions.

03

Browser-in-the-Browser

Fake login window overlay

What it is

BitB attacks render a fake browser window inside the webpage using HTML and CSS. The fake window includes a convincing URL bar showing a trusted domain (accounts.google.com), window control buttons, and a login form with password fields. Because the entire "popup" is part of the page DOM, the URL bar is attacker-controlled — but it looks identical to a real browser popup.

What PhishWatch observes

fake_browser_window_detected

DOM element matching BitB structural patterns: positioned overlay (fixed/absolute, 400–900px dimensions) containing a fake URL bar with trusted domain text, window control characters (×/✕), password fields, and SSO branding. Scored across 6 independent indicators — threshold of 80 for HIGH risk, 60 for MEDIUM.

Detection uses a MutationObserver to catch dynamically injected fake windows — the real-world attack pattern where the fake popup appears only after the user clicks "Sign in."

Adversary-in-the-Middle

Reverse-proxy credential relay (Tycoon 2FA, Evilginx)

What it is

AiTM phishing uses a reverse proxy to sit between the user and the real login service. The user sees a pixel-perfect copy of the real login page, enters their credentials, completes MFA — and the proxy captures the session cookie in real time. This bypasses MFA entirely because the attacker obtains the authenticated session, not just the password. Tools like Evilginx, Modlishka, and commercial kits like Tycoon 2FA have industrialized this technique.

Tycoon 2FA Takedown (March 4, 2026): Europol, Microsoft, Cloudflare, and coalition partners dismantled Tycoon 2FA — one of the most prolific AiTM phishing-as-a-service platforms ever observed. At its peak, Tycoon 2FA accounted for approximately 62% of all phishing attempts Microsoft blocked. The service had over 2,000 operators, 30,000+ phishing domains, and was linked to 96,000+ victims worldwide. Even with the takedown, the underlying AiTM technique persists — new kits will emerge. PhishWatch's detection targets the structural mechanics of the attack class, not any specific kit.

Attack flow

1. Proxy pageLooks identical to real login (M365, Gmail)
2. Credentials enteredRelayed to real service in real time
3. PhishWatch detectsForm action origin ≠ page origin + cross-origin POST
4. Session stolenProxy captures session cookie after MFA

What PhishWatch observes

aitm_pattern_correlation

Two independent mechanical signals in the same observation window: (1) form action origin does not match the page origin, and (2) cross-origin POST request observed during the credential submission window. Both conditions must be present — single-signal matches are logged as research signals only.

This two-signal requirement is critical for avoiding false positives. Legitimate federated SSO flows (SAML, OpenID Connect) may exhibit cross-origin behavior, but the form action origin typically matches the identity provider's domain. AiTM proxies create the structural mismatch that PhishWatch detects.

05

Fake Update

SocGholish-style browser update impersonation

What it is

Fake Update pages impersonate browser update dialogs to trick users into downloading malware. The page displays browser branding (Chrome, Firefox, Edge), urgent update language ("Critical update required"), and an executable download link. Real browser updates never come from websites — they happen silently through the browser's built-in update mechanism.

This technique is commonly associated with the SocGholish framework, which compromises legitimate websites to serve fake update prompts to visitors.

What PhishWatch observes

detectFakeUpdatePage

Page content scan detecting three categories of signals: (1) browser brand impersonation (Google Chrome, Mozilla Firefox, Microsoft Edge), (2) update urgency language ("critical update required," "browser out of date," "security update"), and (3) executable download links or update buttons. Requires browser brand + at least one additional signal. Pages on actual browser vendor domains are excluded.

06

AI Lure

Fake AI service impersonation

What it is

AI Lure attacks impersonate popular AI services — ChatGPT, Claude, Gemini, Copilot — combined with ClickFix or ConsentFix social engineering. Attackers register domains that mimic AI service branding and present fake verification prompts that instruct users to run malicious commands or paste OAuth codes. The AI branding provides trust; the social engineering mechanic delivers the payload.

What PhishWatch observes

detectAILurePage

Combination of AI brand presence in page text (ChatGPT, OpenAI, Claude+Anthropic, Gemini+Google, Copilot+Microsoft) with ClickFix-style social engineering (Win+R instruction, paste run dialog, terminal paste) or ConsentFix-style lures (sign-in + paste code). Requires AI brand + at least one social engineering signal. Legitimate AI vendor domains are excluded.

07

Typosquatting

Brand impersonation via domain similarity

What it is

Typosquatting domains mimic major brands through character substitution (arnazon.com), homoglyph swaps (using Cyrillic 'а' instead of Latin 'a'), adjacent-key typos, and other edit-distance variations. Attackers register these domains to host credential harvesting pages. KnowBe4's research documented post-Scattered Spider campaigns using spoofed domains like "marksandspencr.com" to impersonate M&S following the April 2025 breach.

What PhishWatch observes

typosquatting_detected

Backend Levenshtein distance analysis against 500+ curated major brand domains. Domains with low edit distance from known brands, combined with credential keywords in the URL path, are flagged. This is a backend-assisted detection — requires the Render backend.

Newly Registered Domain

Infrastructure age as risk signal

What it is

Domains registered within the last 30 days are disproportionately used for phishing, malware distribution, and social engineering. Legitimate businesses rarely serve credential-sensitive pages from brand-new domains. Research from Krebs on Security shows that most parked domains now serve malicious content, making domain age an increasingly valuable risk signal.

What PhishWatch observes

newly_registered_domain

Domain age checked via VirusTotal API in real time. Domains registered within 30 days are flagged. Results are cached in-memory for 24 hours to avoid redundant API calls. This is a backend-assisted detection. VirusTotal integration is optional — if no API key is configured, or if the call fails, the signal is omitted and navigation proceeds normally (fail-open). Commercial API terms apply for production use.
